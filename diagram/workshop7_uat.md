# Workshop #7: การออกแบบและดำเนินการทดสอบ User Acceptance Testing (UAT)

**ชื่อระบบ:** ระบบร้านหนังสือออนไลน์ (Online Bookstore)  
**วันที่ทดสอบ:** 22 กรกฎาคม 2026  
**ผู้จัดทำ:** ทีมพัฒนาระบบร้านหนังสือออนไลน์  
**เวอร์ชันระบบ:** v1.0.0

## ขั้นตอนที่ 1: วิเคราะห์ Persona (User Persona Analysis)

ระบบร้านหนังสือออนไลน์มีกลุ่มผู้ใช้งานหลัก **3 กลุ่ม** ดังนี้

| # | Persona | ชื่อกลุ่ม | บทบาทและความรับผิดชอบ |
|---|---------|-----------|----------------------|
| 1 | 👤 **Customer** | ลูกค้า / ผู้ซื้อหนังสือ | ผู้ใช้งานทั่วไปที่เข้าสู่ระบบ Storefront เพื่อค้นหาหนังสือ เพิ่มสินค้าลงตะกร้า ชำระเงิน และติดตามสถานะคำสั่งซื้อ รวมถึงจัดการข้อมูลโปรไฟล์ของตนเอง |
| 2 | 🔧 **Staff** | พนักงาน / เจ้าหน้าที่คลังสินค้า | ผู้ใช้งานที่มีสิทธิ์เข้าถึง Staff Portal เพื่อตรวจสอบและจัดการสต็อกหนังสือ (Inventory Management) อัปเดตจำนวนสินค้าคงเหลือ และตรวจสอบสถานะออเดอร์ |
| 3 | 👑 **Admin** | ผู้ดูแลระบบ | ผู้ใช้งานที่มีสิทธิ์สูงสุด เข้าถึง Admin Portal เพื่อจัดการข้อมูลหนังสือ (เพิ่ม/แก้ไข/ลบ), จัดการหมวดหมู่, จัดการบัญชีผู้ใช้ทั้งหมด และดูภาพรวมของระบบ |

**คำอธิบายระดับความสำคัญ:**
- 🔴 **Critical** — ส่งผลกระทบโดยตรงต่อการทำงานหลักของระบบ ต้องแก้ไขก่อนนำไปใช้งานจริง
- 🟠 **High** — ฟีเจอร์สำคัญที่ขาดหายหรือทำงานบกพร่อง ควรแก้ไขในเวอร์ชันถัดไปโดยเร็ว
- 🟡 **Medium** — ส่งผลกระทบต่อ UX แต่ยังมีทางเลือกอื่นทดแทนได้
- 🟢 **Low** — ปัญหาเล็กน้อยที่ไม่กระทบการทำงานหลัก

### 4.3 สรุปผลการประเมิน UAT (UAT Evaluation Summary)

จากการดำเนินการทดสอบ User Acceptance Testing สำหรับ **ระบบร้านหนังสือออนไลน์ (Online Bookstore) v1.0.0** จำนวนทั้งสิ้น **11 Test Cases** ครอบคลุมผู้ใช้งาน 3 กลุ่ม (Customer, Staff, Admin) พบว่ามีจำนวน Test Case ที่**ผ่านการทดสอบ 8 รายการ** และ**ไม่ผ่าน 3 รายการ** คิดเป็น**อัตราการผ่านการทดสอบโดยรวม 72.7%**

ฟังก์ชันหลักของระบบที่ทำงานได้อย่างถูกต้อง ได้แก่ การลงทะเบียนบัญชีผู้ใช้, การค้นหาและดูรายละเอียดหนังสือ, การเพิ่มสินค้าลงตะกร้าและ Checkout, การเข้าสู่ระบบ Staff Portal พร้อมระบบ Role-based Access Control, การตรวจสอบสต็อก, และการจัดการข้อมูลหนังสือ (เพิ่ม/แก้ไข/ลบ) โดย Admin

อย่างไรก็ตาม พบปัญหาที่ต้องได้รับการแก้ไข **3 รายการ** ก่อนนำระบบไปใช้งานจริงในสภาพแวดล้อม Production ดังนี้:

**ปัญหาระดับ Critical (ต้องแก้ไขทันที):** ISS-002 เรื่องการ Persist สต็อกสินค้าลงฐานข้อมูลจาก Staff Portal เป็นปัญหาที่มีความรุนแรงมากที่สุด เนื่องจากหากข้อมูลคงคลังสินค้าไม่ถูกบันทึกอย่างถาวร อาจทำให้เกิดสินค้า Oversell และส่งผลเสียต่อธุรกิจโดยตรง

**ปัญหาระดับ High (ควรแก้ไขโดยเร็ว):** ISS-001 และ ISS-003 คือการขาด UI สำหรับดูรายละเอียดออเดอร์ของลูกค้า และการที่ Admin ยังไม่สามารถจัดการ Role ผู้ใช้ผ่านหน้าเว็บได้ ทั้งสองฟีเจอร์ส่งผลต่อ User Experience และประสิทธิภาพการดำเนินงาน

**ข้อเสนอแนะ:** ทีมพัฒนาควรดำเนินการแก้ไข ISS-002 เป็นลำดับแรกโดยตรวจสอบ API Endpoint สำหรับ Inventory Update และยืนยันว่า PATCH/PUT request ถูกส่งไปยัง Backend อย่างถูกต้อง จากนั้นดำเนินการสร้างหน้า Order Detail สำหรับลูกค้า และพัฒนาฟังก์ชัน Role Management สำหรับ Admin ตามลำดับความสำคัญ หลังจากแก้ไขครบทั้ง 3 ประเด็นแล้ว ควรดำเนินการ Regression Testing ก่อน Deploy เวอร์ชันถัดไป
