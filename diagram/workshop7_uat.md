# Workshop #7: การออกแบบและดำเนินการทดสอบ User Acceptance Testing (UAT)

**ชื่อระบบ:** ระบบร้านหนังสือออนไลน์ (Online Bookstore)
**วันที่ทดสอบ:** 22 กรกฎาคม 2026
**ผู้จัดทำ:** ทีมพัฒนาระบบร้านหนังสือออนไลน์
**เวอร์ชันระบบ:** v1.0.0
**เทคโนโลยี:** Next.js 16, React 19, MySQL, Tailwind CSS v4, JWT Authentication

## เกณฑ์การประเมินกิจกรรม (Rubric Assessment)

| เกณฑ์ | ระดับ 1 (5 คะแนน) | ระดับ 2 (7 คะแนน) | ระดับ 3 (10 คะแนน) |
|---|---|---|---|
| ความครบถ้วนของ TEST CASE | ครอบคลุมน้อยกว่า 50% | ครอบคลุม 50–80% | ครอบคลุมฟังก์ชันหลักทั้งหมด 100% |
| การอ้างอิง PERSONA | ไม่ระบุ Persona | ระบุบางส่วน | ครบทุก Persona พร้อมบทบาทชัดเจน |
| EXPECTED RESULT | ไม่ชัดเจน วัดผลไม่ได้ | ค่อนข้างครบถ้วน | ชัดเจน วัดผลได้ ครบถ้วน |
| หลักฐานการทดสอบ (Actual Result) | ไม่มีหลักฐาน | มีบางส่วน | ครบถ้วน ระบุผลจริงที่พบ |
| สรุปผลและ ISSUE Report | ไม่มี | สรุปรวม ไม่แยก Issue | วิเคราะห์ครบ แยก Issue + ระดับความสำคัญ |

## ขั้นตอนที่ 1: วิเคราะห์ Persona (User Persona Analysis)

ระบบร้านหนังสือออนไลน์มีกลุ่มผู้ใช้งานหลัก **3 กลุ่ม** ดังนี้

| # | Persona | ชื่อกลุ่ม | จุดเข้าถึงระบบ | บทบาทและความรับผิดชอบหลัก |
|---|---------|-----------|--------------|---------------------------|
| 1 | 👤 **Customer** | ลูกค้า / ผู้ซื้อหนังสือ | `/` , `/browse` , `/cart` , `/checkout` , `/orders` | ค้นหาและเลือกซื้อหนังสือ, เพิ่มสินค้าลงตะกร้า, ชำระเงินผ่านการโอนเงิน/สแกน QR, แนบหลักฐานสลิป, ติดตามสถานะคำสั่งซื้อ, จัดการโปรไฟล์ |
| 2 | 🔧 **Staff** | พนักงาน / เจ้าหน้าที่คลังสินค้า | `/staff` , `/staff/inventory` , `/staff/orders` | ตรวจสอบและอนุมัติหลักฐานชำระเงิน, เปลี่ยนสถานะออเดอร์, ตรวจสอบ/อัปเดตจำนวนสต็อกสินค้า, รับแจ้งเตือนสินค้าใกล้หมด |
| 3 | 👑 **Admin** | ผู้ดูแลระบบ | `/admin` , `/admin/books` , `/admin/categories` , `/admin/users` | จัดการข้อมูลหนังสือทั้งหมด (CRUD), จัดการหมวดหมู่, จัดการ Role ผู้ใช้, ดู Dashboard รายงานยอดขายและสถิติระบบ |

### Q1: หากลูกค้ากดปุ่มชำระเงินซ้ำ 5 ครั้ง ระบบจะป้องกันอย่างไร?
**แนวทางตอบ:**
- **Frontend:** ปุ่ม "ยืนยันคำสั่งซื้อ" จะถูก `Disabled` ทันทีหลังกดครั้งแรก พร้อมแสดง Loading Spinner ป้องกัน Double-click
- **Backend:** ใช้หลักการ **Idempotency** — ตรวจสอบว่ามีออเดอร์ที่ยังประมวลผลอยู่จาก Session เดียวกันหรือไม่ หากมีการส่ง Request ซ้ำ ระบบตอบ `409 Conflict` แทนการสร้างออเดอร์ซ้ำ

### Q2: หากชำระเงินสำเร็จแต่ Database ล้มกลางทาง ข้อมูลจะสูญหายหรือไม่?
**แนวทางตอบ:**
- ระบบใช้ **Database Transaction (ACID)** — หาก INSERT ออเดอร์ หรือ UPDATE สต็อกล้มเหลวขั้นตอนใดขั้นตอนหนึ่ง ระบบจะ **Rollback** ทันที ไม่มีข้อมูลบันทึกครึ่งเดียว
- สำหรับ Webhook จากระบบชำระเงินภายนอก จะมี **Audit Log** บันทึก Payload ไว้ สามารถ Re-process ได้เมื่อ DB กลับมา Online

### Q3: หากสินค้าหมดสต็อกระหว่างที่ลูกค้ากำลัง Checkout ระบบทำอย่างไร?
**แนวทางตอบ:**
- Backend ทำ **Stock Re-verification แบบ Row-level Lock** (`SELECT ... FOR UPDATE`) ก่อนสร้างออเดอร์ทุกครั้ง
- หากสต็อกไม่พอ → ระบบไม่สร้างออเดอร์ ตอบกลับ Error พร้อมแจ้งลูกค้าว่า "สินค้าบางรายการหมดสต็อก" และปรับจำนวนในตะกร้าให้อัตโนมัติ

### Q4: ระบบป้องกัน Unauthorized Access อย่างไร?
**แนวทางตอบ:**
- ใช้ **Next.js Middleware** ตรวจสอบ JWT Token ทุก Request ก่อนเข้าถึงหน้าที่ต้องการ Authentication
- ตรวจสอบ **Role** จาก JWT Payload: `/admin/*` ต้องการ Role `admin`, `/staff/*` ต้องการ Role `staff` หรือ `admin`
- หาก Token หมดอายุหรือผิดปกติ → Redirect ไปหน้า `/login` พร้อมลบ Cookie

### Q5: ทดสอบกรณีผิดปกติ (Negative Test Case) อะไรบ้าง?
**แนวทางตอบ:** ทีมทดสอบเพิ่มเติม Negative Cases ดังนี้
1. **สมัครสมาชิก Email ซ้ำ** → ระบบแสดง Error "อีเมลนี้ถูกใช้งานแล้ว" ไม่สร้างบัญชีซ้ำ
2. **เข้าสู่ระบบด้วย Password ผิด** → แสดง Error "อีเมลหรือรหัสผ่านไม่ถูกต้อง"
3. **ใส่จำนวนสินค้าในตะกร้าเป็นค่าลบ** → ระบบ Reset เป็น 1 อัตโนมัติ
4. **เข้าถึง `/admin` โดยไม่มี Token** → Middleware Redirect ไปหน้า `/login`
5. **Customer พยายามเข้า `/staff`** → Middleware ตรวจ Role แล้ว Redirect ปฏิเสธการเข้าถึง
6. **Admin เพิ่มหนังสือโดยไม่กรอก Title** → Form Validation แสดง Error ก่อนส่ง Request
