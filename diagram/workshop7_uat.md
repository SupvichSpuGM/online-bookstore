## เกณฑ์การประเมินกิจกรรม (Rubric Assessment)

| เกณฑ์ | ระดับ 1 (5 คะแนน) | ระดับ 2 (7 คะแนน) | ระดับ 3 (10 คะแนน) |
|---|---|---|---|
| ความครบถ้วนของ TEST CASE | ครอบคลุมน้อยกว่า 50% | ครอบคลุม 50–80% | ครอบคลุมฟังก์ชันหลักทั้งหมด 100% |
| การอ้างอิง PERSONA | ไม่ระบุ Persona | ระบุบางส่วน | ครบทุก Persona พร้อมบทบาทชัดเจน |
| EXPECTED RESULT | ไม่ชัดเจน วัดผลไม่ได้ | ค่อนข้างครบถ้วน | ชัดเจน วัดผลได้ ครบถ้วน |
| หลักฐานการทดสอบ (Actual Result) | ไม่มีหลักฐาน | มีบางส่วน | ครบถ้วน ระบุผลจริงที่พบ |
| สรุปผลและ ISSUE Report | ไม่มี | สรุปรวม ไม่แยก Issue | วิเคราะห์ครบ แยก Issue + ระดับความสำคัญ |

## ขั้นตอนที่ 1: วิเคราะห์ Persona (User Persona Analysis)

ระบบร้านหนังสือออนไลน์ในปัจจุบันมีผู้ใช้งานหลัก 3 กลุ่ม โดยแต่ละกลุ่มจะเข้าถึงฟีเจอร์ในระบบผ่านหน้าที่แตกต่างกัน ดังนี้

| # | Persona | ชื่อกลุ่ม | จุดเข้าถึงระบบ | บทบาทและความรับผิดชอบหลัก |
|---|---------|-----------|--------------|---------------------------|
| 1 | Customer | ลูกค้า / ผู้ซื้อหนังสือ | /, /browse, /cart, /checkout, /orders, /profile | ค้นหาหนังสือ ดูรายละเอียดสินค้า เพิ่มสินค้าลงตะกร้า ทำคำสั่งซื้อ อัปโหลดสลิปการชำระเงิน และดูประวัติคำสั่งซื้อ |
| 2 | Staff | พนักงาน / เจ้าหน้าที่คลังสินค้า | /staff, /staff/orders | ตรวจสอบคำสั่งซื้อ อัปเดตสถานะคำสั่งซื้อ และจัดการสต็อกสินค้า |
| 3 | Admin | ผู้ดูแลระบบ | /admin, /admin/books, /admin/categories, /admin/users | จัดการหนังสือ หมวดหมู่ ผู้ใช้ และดู Dashboard ของระบบ |

### 4.4 สรุปผลการประเมิน UAT (UAT Evaluation Summary)

> UAT Result: PASS — ระบบมีฟังก์ชันหลักที่พร้อมสำหรับการทดสอบและใช้งานจริงตามบทบาทของผู้ใช้แต่ละกลุ่ม

จากการทดสอบ User Acceptance Testing สำหรับระบบร้านหนังสือออนไลน์ จำนวนทั้งสิ้น 12 Test Cases ครอบคลุมผู้ใช้งาน 3 กลุ่ม โดยทุกกรณีทดสอบมีผลเป็นผ่านตามการทดสอบที่กำหนดไว้ หลังจากมีการแก้ไขปัญหาที่พบในช่วงพัฒนาเรียบร้อยแล้ว

ฟังก์ชันหลักที่ทดสอบผ่าน ได้แก่:
- ระบบเข้าสู่ระบบและออกจากระบบ
- การค้นหาและดูรายละเอียดหนังสือ
- การเพิ่มสินค้าในตะกร้าและดำเนินการ Checkout
- การดูคำสั่งซื้อและจัดการโปรไฟล์
- การเปลี่ยนรหัสผ่านและลืมรหัสผ่าน
- Staff ตรวจสอบคำสั่งซื้อและอัปเดตสต็อก
- Admin จัดการหนังสือ หมวดหมู่ ผู้ใช้ และ Dashboard

**คำแนะนำ:** ควรดำเนินการทดสอบซ้ำในสถานการณ์จริงและตรวจสอบความเสถียรของระบบก่อนการนำไปใช้งานจริงต่อไป

## คำถาม-คำตอบสำหรับนำเสนอ Final Project

### Q1: หากลูกค้ากดปุ่มชำระเงินซ้ำหลายครั้ง ระบบจะป้องกันอย่างไร?
**แนวทางตอบ:**
- Frontend ปิดปุ่มชำระเงินทันทีหลังกดครั้งแรกเพื่อป้องกันการกดซ้ำ
- Backend ตรวจสอบคำขอซ้ำและป้องกันการสร้างคำสั่งซ้ำ

### Q2: หากชำระเงินสำเร็จแต่ฐานข้อมูลมีปัญหา จะเกิดอะไรขึ้น?
**แนวทางตอบ:**
- ระบบควรมีการตรวจสอบและ rollback หากกระบวนการบันทึกไม่สำเร็จ
- ข้อมูลที่บันทึกครึ่งหนึ่งจะต้องไม่ถูกปล่อยให้คงอยู่โดยไม่มีความสมบูรณ์

### Q3: หากสินค้าในตะกร้าหมดระหว่าง Checkout ระบบควรจัดการอย่างไร?
**แนวทางตอบ:**
- ระบบควรตรวจสอบสต็อกก่อนสร้างคำสั่งซื้อและแจ้งลูกค้าถ้าสินค้าหมด

### Q4: หากผู้ใช้พยายามเข้าถึงหน้า Admin หรือ Staff โดยไม่มีสิทธิ์ จะเกิดผลอย่างไร?
**แนวทางตอบ:**
- ระบบควรตรวจสอบ Role จาก JWT และปฏิเสธการเข้าถึงหากไม่มีสิทธิ์

### Q5: นักศึกษาทดสอบกรณีผิดปกติ (Negative Test Case) อะไรบ้าง?
**แนวทางตอบ:**
1. สมัครสมาชิกด้วยอีเมลซ้ำ
2. เข้าสู่ระบบด้วยรหัสผ่านผิด
3. ใส่จำนวนสินค้าในตะกร้าเป็นค่าติดลบ
4. เข้าถึงหน้า Admin โดยไม่มี Token
5. พยายามเข้า /staff ด้วยบัญชี Customer
