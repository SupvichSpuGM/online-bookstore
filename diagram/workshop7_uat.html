RUBRIC

เกณฑ์การประเมินกิจกรรม

ระดับ 3 — สมบูรณ์ 10 คะแนน

ครอบคลุมฟังก์ชันหลักทั้งหมด 100% ครบทุก Persona มี Expected/Actual Result ชัดเจน สรุปผลและ Issue ครบถ้วนพร้อมระดับความสำคัญ

ระดับ 2 — ค่อนข้างครบ 7 คะแนน

ครอบคลุม 50–80% มีหลักฐานบางส่วน Expected Result ไม่ครบทุก Test Case สรุป Issue แต่ขาดระดับความสำคัญ

ระดับ 1 — ต้องปรับปรุง 5 คะแนน

ครอบคลุมน้อยกว่า 50% ไม่ระบุ Persona ชัดเจน ไม่มีหลักฐาน ไม่มีสรุปผลหรือ Issue Report

DEFENSIVE Q&A

คำถาม-คำตอบสำหรับนำเสนอ Final Project

Q1หากลูกค้ากดปุ่มชำระเงินซ้ำ 5 ครั้ง ระบบจะป้องกันอย่างไร?
  • Frontend: ปุ่ม "ยืนยันคำสั่งซื้อ" จะถูก Disabled ทันทีหลังกดครั้งแรก พร้อมแสดง Loading Spinner ป้องกัน Double-click / Spam
  • Backend (Idempotency): ตรวจสอบว่ามีออเดอร์ที่ยังประมวลผลอยู่จาก Session เดียวกันหรือไม่ หากมี Request ซ้ำ ระบบตอบ 409 Conflict แทนการสร้างออเดอร์ซ้ำ
Q2หากชำระเงินสำเร็จแต่ Database ล้มกลางทาง ข้อมูลจะสูญหายหรือไม่?
  • Database Transaction (ACID): ทุก Operation (INSERT Order + UPDATE Stock) อยู่ใน Transaction เดียวกัน หากขั้นตอนใดล้มเหลว ระบบ Rollback ทันที ไม่มีข้อมูลบันทึกครึ่งทาง
  • Audit Log: สำหรับ Webhook ชำระเงินภายนอก จะมี Log บันทึก Payload ไว้ สามารถ Re-process ได้เมื่อ DB กลับมา Online โดยไม่สูญหาย
Q3หากสินค้าหมดสต็อกระหว่างที่ลูกค้ากำลัง Checkout จะเกิดอะไรขึ้น?
  • Backend ทำ Stock Re-verification แบบ Row-level Lock (SELECT ... FOR UPDATE) ก่อนสร้างออเดอร์ทุกครั้ง
  • หากสต็อกไม่พอ → ระบบไม่สร้างออเดอร์ ตอบ Error กลับ Frontend แจ้งลูกค้าว่า "สินค้าบางรายการหมดสต็อก" และ Auto-adjust จำนวนในตะกร้า
Q4ระบบป้องกัน Unauthorized Access (การเข้าถึงโดยไม่มีสิทธิ์) อย่างไร?
  • Next.js Middleware: ตรวจสอบ JWT Token ใน Cookie ทุก Request ก่อนเข้าหน้าที่ต้องการ Authentication
  • Role-based Access Control (RBAC): /admin/* ต้องการ Role admin, /staff/* ต้องการ Role staff หรือ admin
  • หาก Token หมดอายุหรือ Role ไม่ตรง → Redirect ไปหน้า /login พร้อมลบ Cookie
Q5นักศึกษาทดสอบกรณีผิดปกติ (Negative Test Cases) อะไรบ้าง?
  • Email ซ้ำ: สมัครด้วย Email ที่มีในระบบ → ระบบแสดง Error "อีเมลนี้ถูกใช้งานแล้ว" ไม่สร้างบัญชีซ้ำ
  • Password ผิด: Login ด้วย Password ผิด → แสดง "อีเมลหรือรหัสผ่านไม่ถูกต้อง"
  • จำนวนสินค้าเป็นลบ: ป้อนจำนวนในตะกร้าเป็น -1 → ระบบ Reset เป็น 1 อัตโนมัติ
  • ไม่มี Token เข้า Admin: เข้า /admin โดยตรง → Middleware ดักจับ Redirect ไป /login
  • Customer เข้า Staff: Customer พยายามเข้า /staff → Middleware ตรวจ Role แล้วปฏิเสธ
  • Form ไม่ครบ: Admin เพิ่มหนังสือโดยไม่กรอก Title → Form Validation แสดง Error ก่อนส่ง Request
Q6ระบบรองรับการใช้งานพร้อมกันหลายคน (Concurrency) ได้อย่างไร?
  • Row-level Locking: ใช้ SELECT ... FOR UPDATE เมื่อ Checkout เพื่อป้องกัน Race Condition กรณีลูกค้าหลายคนสั่งสินค้าที่เหลือ 1 ชิ้นพร้อมกัน
  • API Stateless Design: API ทุก Endpoint Stateless รองรับ Horizontal Scaling ได้ในอนาคต
  • Connection Pool: ใช้ MySQL Connection Pool ป้องกัน Connection exhaustion เมื่อมี Traffic สูง
