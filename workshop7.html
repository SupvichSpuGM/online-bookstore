หลักการ UAT (User Acceptance Testing): การทดสอบแบบยอมรับโดยผู้ใช้เพื่อตรวจสอบว่าระบบสามารถรองรับขั้นตอนการทำงานจริง (Business Workflows) ตามประเภทผู้ใช้ที่แตกต่างกันได้ครบถ้วน โดยการตรวจสอบจะดำเนินการในรูปแบบ Manual Testing เพื่อประเมินความสมบูรณ์ ความพร้อม และวิเคราะห์ความปลอดภัยของระบบก่อนส่งมอบ Go-Live

ในแบบประเมินนี้ทีมงานได้กำหนดผู้ใช้หรือ Persona หลัก 3 บทบาท คือ ลูกค้า (Customer), พนักงานคลังสินค้า (Staff), และ ผู้จัดการ/ผู้บริหาร (Manager) และทำการทดสอบในขอบข่ายการทำงานจริงทั้งหมด 12 รายการ ซึ่งมีกรณีทดสอบผ่าน 7 รายการ และไม่ผ่าน 5 รายการ (คิดเป็น 58.33% Pass Rate) โดยข้อผิดพลาดทั้งหมดจะถูกจัดทำเป็นบันทึกสรุปปัญหา (Issue Registry) เพื่อวางแผนแก้ไขระบบต่อไป

แนวทางการตอบคำถามช่วงนำเสนอ Final Project (QA Preparation)

แนวทางการชี้แจงปัญหาและกลไกความปลอดภัยเชิงสถาปัตยกรรมระบบ เพื่อใช้สำหรับตอบข้อสงสัยของคณะกรรมการผู้ประเมินโครงงาน:

💬 คำถามที่ 1: หากลูกค้ากดชำระเงินซ้ำๆ 5 ครั้ง ติดต่อกัน ระบบจะป้องกันอย่างไร? ▶
คำตอบเชิงเทคนิค (Technical Answer):
1. ฝั่งหน้าบ้าน (Frontend): ทันทีที่ผู้ใช้คลิกปุ่ม "ชำระเงิน" สคริปต์ JavaScript จะสั่งเปลี่ยนคุณสมบัติปุ่มให้เป็น disabled ทันทีเพื่อปิดกั้นไม่ให้กดซ้ำ พร้อมทั้งแสดง UI Loading Spinner
2. ฝั่งหลังบ้าน (Backend): นำเทคนิค Idempotency Key มาร่วมใช้งาน โดยสร้างโทเค็นเฉพาะของธุรกรรมนั้นๆ แนบไปกับคำขอ (Request Headers) และใช้หน่วยความจำความเร็วสูงอย่าง **Redis** เป็นกลไกช่วยทำ Distributed Lock (เช่น ตั้งเวลา Lock 15 วินาที) หากมีคำสั่งซื้อที่มีโทเค็นเดียวกันส่งซ้ำซ้อนเข้ามา ระบบเซิร์ฟเวอร์จะปฏิเสธการประมวลผลธุรกรรมที่เกินมาและส่ง HTTP Status 409 Conflict กลับไปทันที
💬 คำถามที่ 2: หากการชำระเงินสำเร็จ (ยอดหักจากบัญชีแล้ว) แต่ Database เกิดล่มระหว่างนั้น จะเกิดอะไรขึ้นและจะรับมืออย่างไร? ▶
คำตอบเชิงเทคนิค (Technical Answer):
1. การประยุกต์ใช้ Database Transaction (ACID): การจัดการบันทึกข้อมูลจะอยู่ภายใต้การสแกน Transaction บล็อกเดียวกัน เช่น START TRANSACTION และส่งกระบวนการบันทึกสถานะชำระเงินคู่ไปกับการหักลดสต็อก หากส่วนใดส่วนหนึ่งพังลงและยกเลิกก่อนการทำ COMMIT ตัวระบบ SQL จะยกเลิกสถานการณ์ที่ค้างไว้ทิ้ง (Rollback) เพื่อไม่ให้ระบบเกิดภาวะข้อมูลขัดแย้ง
2. การใช้ระบบ Event Queue / Webhook Retry: ทางเดินของข้อมูลผ่าน Payment Gateway จะใช้โครงสร้าง Webhook Receiver ซึ่งจะมี Retry Policy ส่งสัญญาณเตือนซ้ำ หากฝั่ง Gateway ไม่ได้รับสัญญาณ HTTP 200 OK ตอบกลับ (เพราะ DB ล่ม) นอกจากนั้นระบบจะเขียนบันทึกประวัติธุรกรรมแบบสำรองภายนอก (Audit Logs File) เป็นไฟล์สำรองเพื่อใช้สำหรับมาไล่สอบทานยอดเงินและกู้คืน (Manual Reconciliation) ภายหลังได้อย่างแม่นยำ
💬 คำถามที่ 3: หากสินค้าหมดระหว่างที่ลูกค้ากำลังทำรายการเช็คเอาท์ (Checkout) ระบบควรตอบสนองอย่างไร? ▶
คำตอบเชิงเทคนิค (Technical Answer):
1. **Optimistic Locking:** ในฐานข้อมูล MySQL ตารางสต็อกจะใช้การระบุเงื่อนไขตรวจสอบปริมาณล่าสุด เช่น การส่งคำสั่งแก้ไขสต็อกพร้อมเงื่อนไข UPDATE books SET stock = stock - 1 WHERE id = :id AND stock >= 1 ซึ่งหากระหว่างการจองนั้นไม่มีสินค้าหลงเหลืออยู่ ระบบ SQL จะแจ้งเตือนผลกระทบ 0 แถวกลับมา
2. **Real-time Inventory Re-verification:** เมื่อเรียกใช้ API ยืนยันคำชำระเงิน `/api/orders/checkout` ระบบ backend จะดึงสเตตัสความถูกต้องของสต็อกมาเปรียบเทียบในตะกร้าอีกครั้งก่อนรับชำระ หากมีผู้ใช้ซื้อตัดหน้าไปก่อน ระบบจะส่งข้อมูลตอบกลับปฏิเสธ พร้อมดึงประวัติตะกร้ามาปรับแก้จำนวนให้เป็นจำนวนคงเหลือจริง และเตือนผู้ใช้ว่า "ขออภัย สต็อกสินค้าไม่พอ"
💬 คำถามที่ 4: หากยอดเงินโอนที่ชำระมาไม่ตรงกับยอดเรียกเก็บจริงในระบบ จะมีวิธีการตรวจสอบและสกัดกั้นอย่างไร? ▶
คำตอบเชิงเทคนิค (Technical Answer):
1. **การใช้เศษสตางค์สุ่ม (Unique Decimal Points):** ในโหมดโอนเงินผ่านระบบบัญชีปกติ จะใช้วิธีสุ่มยอดจุดทศนิยมแนบท้ายให้กับลูกค้ารายนั้น เช่น 300.18 บาท เพื่อความแม่นยำในการคัดกรองยอด
2. **เชื่อมต่อ Payment Gateway API เต็มรูปแบบ:** การป้องกันปัญหาที่ดีที่สุดคือการยกเลิกการตรวจสลิปรูปแบบเขียนมือและหันมาประยุกต์ใช้ PromptPay Dynamic QR Code (ผ่านธนาคารโดยตรง) ระบบจะสั่งผลิต QR Code ที่ฟิกซ์จำนวนเงิน ยอดชำระ และข้อมูลคำสั่งซื้อไว้ชัดเจน หากยอดเงินโอนเข้ามาไม่ตรง ยอดเงินจะไม่ผ่านตัว API Callback และหากเกิดข้อสงสัย ระบบจะออโต้โอนย้ายคำสั่งซื้อนั้นไปอยู่ในหมวด "รอการยืนยัน/โอนเงินไม่ตรงกำหนด" พร้อมแจ้งเจ้าหน้าที่เข้ามาแมนนวลเช็คเป็นกรณีพิเศษ
💬 คำถามที่ 5: นักศึกษาได้ทดสอบกรณีผิดปกติ (Negative Test Case) อะไรบ้างในโครงการนี้? ▶
คำตอบเชิงเทคนิค (Technical Answer):
ทีมงานจัดทับการทดสอบ Negative Test Case เพื่อดักบั๊กดังนี้:
1. **ฝั่งสมัครสมาชิก/ความปลอดภัย:** ป้อนรูปแบบ Email ที่ไม่มีฟอร์แมตถูกต้อง, พยายามกรอกรหัสผ่านผิดติดต่อกันเกิน 5 ครั้ง (ระบบล็อก IP บล็อกความปลอดภัยชั่วคราว)
2. **ฝั่งตะกร้าสินค้า:** การลองส่งรหัสตะกร้าด้วยตัวเลขติดลบ (เช่น สั่งซื้อ -2 เล่ม) หรือการยัดอักขระแปลกปลอมผ่าน URL เพื่อเช็คกลไกการปัดค่าออกของเซิร์ฟเวอร์
3. **ระบบจัดการสต็อก:** การสั่งซื้อหนังสือ 1,000 เล่มในขณะที่คลังเหลือ 2 เล่ม เพื่อรับประกันว่า API จะสามารถตอบสนองปฏิเสธความยาวเกินคิวได้อย่างถูกต้องสมบูรณ์
ปรับปรุงผลการคำนวณ UAT สำเร็จ!
